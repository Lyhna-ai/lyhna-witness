Install by agent

Paste this into your MCP-capable agent (Claude Code, Cursor, Codex…). It sets up the connection, runs a test call, and renders your first receipt.

Set up Lyhna so my tool calls produce an AI Work Receipt. Lyhna is an MCP proxy that
wraps an MCP server I already use and witnesses every tool call routed through it.

1. Wrap ONE MCP server I already use (e.g. filesystem) by adding this to my MCP client
   config (e.g. .mcp.json), keeping my existing server as the upstream:

   {
     "mcpServers": {
       "filesystem-witnessed": {
         "command": "npx",
         "args": ["-y", "@lyhna/mcp", "stdio"],
         "env": {
           "LYHNA_PROXY_BIND_MODE": "demo",
           "LYHNA_PROXY_CLAIM_CAPTURE": "1",
           "LYHNA_PROXY_LOOP_ID": "lyhna-setup-test",
           "LYHNA_PROXY_GOAL": "first witnessed run",
           "LYHNA_PROXY_UPSTREAM_COMMAND": "npx",
           "LYHNA_PROXY_UPSTREAM_ARGS_JSON": "[\"-y\",\"@modelcontextprotocol/server-filesystem\",\"/path/I/allow\"]"
         }
       }
     }
   }

   demo mode runs offline with no key (receipts are unsigned). For signed receipts,
   replace LYHNA_PROXY_BIND_MODE=demo with LYHNA_API_KEY=<my beta key>.
   LYHNA_PROXY_CLAIM_CAPTURE=1 turns on claim capture so the receipt can compare what
   I claimed to what was witnessed — without it, the receipt only shows observed calls.

2. Tell me to reload/restart my MCP client so the "filesystem-witnessed" server connects.

3. Once it's connected, record a claim with the record_claim tool describing what you're
   about to do, then make ONE real tool call through "filesystem-witnessed" — a small test
   call — so Lyhna witnesses it. The claim + the witnessed call are what the receipt compares.

4. Export the closed loop with the proxy's export-pack verb to get a witness-input.json,
   then render the AI Work Receipt Capsule with the witness CLI (from a lyhna-witness
   clone):
       node src/cli.mjs <witness-input.json> ./receipt --okf --pam

5. Open ./receipt/CAPSULE.md and show me the verdict and what was witnessed vs. claimed.

If any step needs me (editing config, restarting the client, a beta key), stop and tell me
exactly what to do — don't guess or fake a result.

The last line matters: Lyhna only witnesses calls routed through it. If your agent can't reload its MCP config itself, it will ask you to — that's expected during the beta.

What actually works today

  • The proxy is on npm. npx -y @lyhna/mcp wraps an MCP server you already use. No rewrite, no new platform.
  • Demo mode needs no key. LYHNA_PROXY_BIND_MODE=demo runs fully offline — same flow, same artifacts — but receipts are deliberately unsigned.
  • Signed receipts need a beta key. A LYHNA_API_KEY connects to Lyhna's hosted witness service, which is invite-only during the private beta.
  • The receipt renderer isn't on npm yet. The witness CLI (node src/cli.mjs) runs from a source clone for now — packaging it is part of what the beta is for.
  • No plugin store, no one-command install. Setup is the guided sequence above. We won't pretend otherwise.

How you'll know it's working

Four states tell you exactly where you are.

  • 🟢 Connected

    Your agent's calls to the witnessed server are routing through Lyhna and being recorded.

  • 🟡 Waiting for first witnessed tool call

    Connected, but no call has crossed the boundary yet. Make one test call to see a receipt.

  • 🟢 Test receipt created

    Lyhna rendered your first AI Work Receipt Capsule. Open CAPSULE.md and read the verdict.

  • 🔴 Not connected · outside the witness path

    The agent is calling tools that don't route through Lyhna. Those calls aren't witnessed — only what's routed through the proxy appears on a receipt.
